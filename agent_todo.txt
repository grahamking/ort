## Top 10 TODOs by Value

### 1. **Parse JSON error responses from OpenRouter** (`src/net/http.rs:295`)
**Value: Very High.** Currently when the API returns an error (e.g., "not a valid model ID"), `ort` just dumps the raw JSON blob as a string. Parsing it to extract the `message` and `code` fields would give users clear, actionable error messages instead of opaque JSON dumps. Low effort, immediate UX improvement for every error case.

### 4. **Error handling: try all resolver addresses, not just the first** (`src/net/http.rs:312`)
**Value: Very High.** DNS lookups commonly return multiple addresses. If the first one fails, `ort` gives up entirely instead of trying the next one. This makes the tool fragile against any temporary network issue, load balancer node failure, etc. Simple retry loop would dramatically improve reliability. Related: the function currently hits a "connect error handling TODO" dead-end anyway.
### 5. **Handle multiple parallel tool calls in agent mode** (`src/input/agent.rs:224`)
**Value: Very High.** LLMs increasingly emit multiple independent tool calls in a single response. If `ort` only processes the first one, the agent loop wastes a round-trip for each independent call. Supporting parallel returns would make agent mode significantly faster and more correct.

### 6. **Support non-PDF files (or restrict `-f` to PDF)** (`src/input/to_json.rs:415`)
**Value: High.** File attachments currently hardcode `data:application/pdf;base64` — sending any other file type as PDF will confuse the model. Either expanding to detect MIME types or explicitly restricting to PDF would prevent subtle failures. The more flexible approach (MIME detection) would unlock images, code files, etc.

### 7. **Send reasoning content back in conversation history** (`src/common/data.rs:544,551`)
**Value: High.** The `assistant()` and `assistant_with_tool_call()` constructors discard reasoning content. This means the model's chain-of-thought is lost in subsequent turns of the conversation. Preserving it is essential for models that output reasoning to maintain context quality.

### 8. **Store files in `last.json` so resume works with file attachments** (`src/common/data.rs:394`)
**Value: High.** When you resume a conversation with `ort`, file attachments are lost because they're not persisted in the last-state file. This means you can't use `-` (resume) after attaching files, which is a broken UX path. Storing base64 data in `last.json` would be straightforward if files are small, or storing file paths for larger ones.

### 9. **Support tool parameter arrays** (`src/input/to_json.rs:292`)
**Value: High.** Tool schemas can have `type: "array"` parameters, which need `"items": {...}` in the JSON schema. Without this, any tool with an array parameter (e.g., `list_files(["dir1", "dir2"])`) can't generate correct schemas. Blocks agent tooling capabilities.

### 10. **OrtError must be able to hold a String** (`src/input/args.rs:339`, also `src/common/data.rs:1075`)
**Value: Medium-High.** The error type can only hold `&'static str`, so dynamic error messages (from owned strings) must be printed to stdout and then replaced with "See above". This makes error handling inconsistent and loses context when errors propagate. Fixing this has a cascading benefit across all error handling in the codebase.
